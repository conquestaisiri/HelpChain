## 1. Platform Vision & Mission

**1.1 Mission Statement:**
HelpChain is a social-help coordination platform designed to seamlessly connect individuals in need with helpers, volunteers, and donors. Its mission is to create a trust-driven ecosystem where requesting and providing help is intuitive, safe, fast, and engaging. Every interaction is designed to increase confidence in the system, ensure transparency, and foster a supportive community.

**1.2 Core Principles:**

* **Trust-first:** All users must feel secure when posting or offering help. Verification badges, transparent reviews, and admin moderation reinforce trust.
* **Approachability:** The tone and visual design must be friendly, encouraging, and non-intimidating, even for non-technical users.
* **Professionalism:** The platform must convey credibility through clean layouts, structured content, and consistent typography.
* **Simplicity:** Minimal friction for users; easy to navigate, intuitive forms, clear calls-to-action.
* **Scalability:** Built to grow from Web2 to Web3 with blockchain-based verification in future.

**1.3 Tone & Feel:**

* Friendly like **Duolingo** — empathetic messages, encouraging prompts.
* Professional like **LinkedIn** — polished dashboards, structured layouts.
* Simple like **Airbnb** — uncluttered interface, intuitive flows.
* Modern fintech inspiration for smooth micro-interactions (Venmo, CashApp).

**1.4 Target Audience:**

* Everyday people over 18, non-technical or tech-savvy.
* Volunteers or donors seeking trustworthy ways to help.
* Users valuing clarity, safety, and simple user experience.

**1.5 Goals for Replit AI:**

* Build a visually lively, polished, trustworthy UI.
* Include every micro-interaction, animation, and edge case.
* Provide smooth, intuitive flows from landing page to completed help transaction.
* Implement responsive design and accessibility across devices.
* Ensure modular architecture for future blockchain integration.

## 2. User Roles & Permissions

### 2.1 Help Seekers

**Capabilities:**

* Create help requests with fields:

  * **Title:** Max 100 characters, inline validation.
  * **Description:** Multi-line text, 1,000 character limit, auto-save draft.
  * **Location:** Autocomplete with Google Maps API, validation, fallback manual input.
  * **Type of Help:** Multi-select checkboxes (Financial, Physical Assistance, Supplies, Emotional Support).
  * **Urgency:** Radio buttons (Urgent, Medium, Low) with color-coded badges and animations.
  * **Attachments:** Optional ID, proof of need, images, PDFs; drag-and-drop with preview thumbnails.
* View helper profiles before accepting help, including rating, completed helps, verification badge.
* Real-time chat with helpers who offer assistance.
* Rate helpers post-completion.

**UI/UX Details:**

* Multi-step creation form with **progress bar** (Step 1 of 5, etc.).
* **Inline validation** with animated error icons.
* Success banners slide down with green checkmark animation.
* CTA buttons: bright yellow (#FFB400), hover glow effect.
* Urgency badges pulse if “Urgent” selected.
* Draft auto-save every 30 seconds.
* Form accessible via keyboard navigation and screen readers.

**Edge Cases:**

* Required fields left empty → inline error.
* Attachment exceeds 5MB → prompt error.
* Location autocomplete fails → manual entry fallback.
* Multiple submissions blocked for duplicate requests.

### 2.2 Helpers/Donors

**Capabilities:**

* Browse help requests with filters:

  * Location (map view optional), type, urgency, verified seekers only.
  * Sort by newest, nearest, most urgent.
* Click “Offer Help” → opens **real-time chat modal** with prefilled template message.
* Upload proof of assistance: images, PDF documents.
* Post-assistance rating of seeker.

**UI/UX Details:**

* Feed cards: hover lift, shadow, subtle scale.
* Filters sidebar: collapsible on desktop, dropdown on mobile.
* Sorting options with intuitive icons.
* CTA buttons: green (#3BB273), hover glow.
* Map integration: optional pin drop, click to view requests near you.

**Edge Cases:**

* Multiple helpers click “Offer Help” simultaneously → queue or modal conflict resolution.
* File upload fails → retry mechanism.
* User attempts to offer help to own request → prevent and alert.

### 2.3 Admins

**Capabilities:**

* Approve/deny help requests before public listing.
* Monitor disputes flagged by users.
* Block malicious users or verify identities.
* Access dashboard metrics: total users, requests, helpers, completed helps, flagged incidents.
* Manage system-wide settings: email templates, notification preferences.

**UI/UX Details:**

* Dashboard with tabs: Requests, Disputes, Reports, Analytics, Settings.
* Graphs for completed requests, top helpers, request locations.
* Approve/deny buttons: color-coded, hover animation.
* CSV export for analytics.
* Responsive grid layout; collapsible sidebar on mobile.

**Edge Cases:**

* Admin deletes verified user → system logs deletion.
* Flagged chat → modal shows full history.
* Dashboard metrics refresh real-time with animation.

### 2.4 Role-Based Access Control

* JWT tokens encode user role.
* Middleware checks endpoints for role authorization.
* Frontend hides UI elements for unauthorized roles.
* Backend rejects unauthorized API calls with 403 Forbidden and logs attempt.

## 3. Authentication & Security

**3.1 Login Options:**

* Email/password.
* Google OAuth.
* Optional: Facebook, Apple in future.

**3.2 Security:**

* Password hashing with bcrypt.
* Server and client-side input validation.
* Rate limiting: max 5 failed login attempts per hour.
* Strong password rules: min 8 characters, 1 uppercase, 1 number, 1 special character.
* Session timeout and JWT refresh strategy.

**3.3 Password Recovery:**

* Email token with 1-hour expiration.
* Clear error messages for expired or invalid tokens.
* Flow: Request reset → Email link → Enter new password → Success banner.

**3.4 Edge Cases:**

* Email exists → prompt “This email is already registered.”
* Login fails → prompt “Credentials don’t match our records.”
* Expired token → redirect to login page with friendly message.

## 5. Backend Architecture

**5.1 Server:**

* Node.js + Express.js.
* REST API endpoints:

  * Auth: register, login, logout, password reset.
  * Requests: CRUD operations.
  * Offers: browse, respond, chat.
  * Messaging: real-time or polling.
  * Reviews: submit & fetch.

**5.2 Database (MongoDB):**

* **Users:** _id, name, displayName, email, passwordHash, role, location, rating, totalHelpGiven, totalHelpReceived, verificationStatus.
* **HelpRequests:** _id, title, description, requesterId, typeOfHelp, urgency, attachments[], status (pending/approved/fulfilled), location, createdAt, updatedAt.
* **Messages:** _id, senderId, receiverId, content (text/image), timestamp, associatedRequestId.
* **Reviews:** _id, reviewerId, revieweeId, rating, comment, associatedRequestId.

**5.3 Security:**

* Input sanitization.
* JWT authentication.
* Rate limiting.
* File upload validation.
* Role-based authorization middleware.
* Audit logs for admin monitoring.

7. Edge Cases & Considerations
7.1 Request Submission Edge Cases

Fake requests: Admin approval required before posting publicly.

Multiple submissions: Duplicate prevention system.

File attachments: Validate size and type, retry failed uploads.

Location errors: Fallback to manual input if autocomplete fails.

7.2 Helper Edge Cases

Multiple helpers responding simultaneously: Queue system and first-come-first-served.

Failure to deliver help: Rating and review system flags for admin attention.

Attempting to help own request → block and alert.

Attachment upload failure → retry or fallback method.

7.3 Messaging Edge Cases

Lost WebSocket connection: Auto-reconnect with exponential backoff.

Message delivery fails → retry mechanism.

Browser notifications blocked → fallback email notifications.

Read receipts failure → retry once, log failure.

7.4 Admin Edge Cases

Admin deletes user → system logs deletion and notifies relevant parties.

Flagged chat → modal with full chat history, timestamped.

Analytics refresh real-time with smooth animation.

Dashboard metrics: fallback default if data missing.

11. Atomic-Level Flows & Interaction Details
11.1 Help Request Creation Flow

Step 1: Enter Title & Description.

Step 2: Select Location (Autocomplete, Map Fallback).

Step 3: Select Type of Help.

Step 4: Attach optional documents/images.

Step 5: Review & Submit.

Success → Admin approval pending → Notifications triggered.

Published → Appears on Feed → Helpers can offer help.

11.2 Offer Help Flow

Helper filters/searches feed.

Click “Offer Help” on request.

Chat modal opens with template message.

Helper confirms commitment → Notification sent to seeker.

Post-help: Upload proof (optional) → Rating & review triggered.

11.3 Admin Approval Flow

Admin sees pending requests → Approve/Deny.

Approve → Request goes live → Notification sent.

Deny → Reason input → Notification sent.

Flagged messages or disputes → Review modal → Action taken.

11.4 Messaging Flow

WebSocket real-time messages.

Typing indicator animation.

Read receipts animation.

Attachment upload → progress bar → success confirmation.

Chat history persistence → scroll to latest.

Notification triggers for new messages.
